## 一、为什么这两个模块是 xiaoli 的灵魂

三个竞品（DeepSeek-TUI、Reasonix、Langcli）做的是「交互式编程代理」——你说一句它做一步。

xiaoli 做的是「自主式编程代理」——你说完需求它就自己去干了。

这中间的差距不是「AI 更聪明」，而是：

> **xiaoli 在动手之前，花足够多的时间把你脑子里模糊的想法，变成一份结构化的、无歧义的、可验证的执行计划。**

完成这个转化的，就是**追问生成器**和**执行计划引擎**。

| 模块 | 做的是 | 类比 |
|---|---|---|
| 追问生成器 | 把模糊需求 → 精确需求 | 医生问诊：「哪里疼？什么时候开始？吃了什么？」 |
| 执行计划引擎 | 把精确需求 → 分步计划 | 手术方案：「先拍CT → 局部麻醉 → 切除 → 缝合」 |

两个模块分开设计，但在交互中是**交替进行**的——用户每回答一轮追问，计划就刷新一次。

## 二、追问生成器

### 2.1 设计目标

> **xiaoli 追问时，用户会觉得「它真的在试图理解我」，而不是「它在走流程」。**

这个感受来自于三个点：

1. **只问通过读代码判断不了的事** — 不问「你用的什么框架」（如果 package.json 里有）
2. **追问有梯度** — 第一轮宏观（目标/约束/边界），第二轮微观（细节/边缘情况），最多三轮
3. **回答不了可以跳过** — 用户说「不知道/不确定」→ 标记为「假设」→ 执行到这一步再问

### 2.2 三级漏斗模型

```
用户输入：一句话任务描述
     │
     ▼
┌──────────────────────────────────────────────────┐
│ 第零级：上下文注入（不提问，只"看"）              │
│ · 读取项目结构（tree）                            │
│ · 读取依赖文件（package.json / pom.xml / pipfile）│
│ · 读取最近修改的源码                              │
│ · 读取 git log（最近10条提交）                    │
│ → 输出：项目快照 JSON（技术栈、模块结构、活跃分支）│
└──────────────────────┬───────────────────────────┘
     │
     ▼
┌──────────────────────────────────────────────────┐
│ 第一级：追问生成（发问）                          │
│ · 基于项目快照 + 用户输入生成追问                 │
│ · 6 大类追问 × 每类最多 1-2 问                   │
│ · 最多 8 个问题，按重要性排序                     │
│ → 输出：追问列表                                  │
└──────────────────────┬───────────────────────────┘
     │
     ▼
┌──────────────────────────────────────────────────┐
│ 第二级：追问收束（梯度追问）                      │
│ · 用户回答后：已确定的 → 锁定，未确定的 → 追问   │
│ · 每轮追问数递减（8→5→3→停）                     │
│ · 用户说「不确定」的直接标记为假设                │
│ · 最多 3 轮（第一轮算一轮）                       │
│ → 输出：完整的问题骨架                            │
└──────────────────────┬───────────────────────────┘
     │
     ▼
  进入执行计划引擎
```

### 2.3 六大追问分类

每次追问必须归类到以下 6 个维度之一。如果某个维度能从项目快照中推断，跳过不问。

| 分类 | 追问方向 | 示例 | 能否推断的触发条件 |
|---|---|---|---|
| **A. 目标定义** | 做什么、做到什么程度、成功标准 | 「导出所有订单还是只导出本月？」 | 用户输入有歧义时 |
| **B. 技术约束** | 用什么、不能用什么、性能要求 | 「前端是否必须用 ant-design-vue？」 | 项目快照能读到的技术栈不问 |
| **C. 范围边界** | 包含什么、不包含什么、影响面 | 「改导出功能会不会影响报表？」 | 涉及共享模块时 |
| **D. 验收标准** | 什么算做完、怎么验证 | 「导出100万条也要3秒内吗？还是可以先做小数据量？」 | 任务涉及性能时 |
| **E. 优先级/紧急度** | 先做什么、后做什么、DDL | 「你们今天必须交付的是导出还是在途追踪？」 | 多个任务同时出现时 |
| **F. 风险/依赖** | 依赖谁、阻碍是什么 | 「这个修改依赖运维改nginx配置吗？」 | 任务涉及外部系统时 |

### 2.4 追问终止策略

满足以下**任意一个条件**即停止追问：

1. **用户明确说「可以开始了」或「先这样做」** — 即使还有未回答的问题
2. **所有未回答的问题都已标记为假设** — 用户可以回答「不确定」
3. **追问轮次达到 3 轮** — 强制结束，未回答的标记为假设
4. **连续两轮没有新增问题** — 信息已饱和

### 2.5 追问交互行为

每个追问在终端里的呈现格式：

```
┌─────────────────────────────────────────────────┐
│ 📋 关于「订单导出优化」我还需要确认 (3/5):       │
│                                                   │
│ 1. [目标] 导出范围：全部订单 or 仅本月？          │
│    └─ 全部，包含已删除的         ✓ 已确认         │
│                                                   │
│ 2. [技术约束] 导出格式：Excel or CSV？             │
│    └─ Excel (.xlsx)              ✓ 已确认         │
│                                                   │
│ 3. [验收标准] 性能基线：当前查询多慢？             │

│                                                   │
│ 4. [风险] 是否涉及生产数据库写入？                 │
│    └─ [等待回答...]                               │

## 三、执行计划引擎

### 3.1 设计目标

> **用户看到执行计划后，心里想的是「这个方案确实能做」，而不是「AI 又在瞎规划」。**

执行计划有三个要求：

1. **可验证** — 每一步有输入/输出定义，执行后能判断「做完没」
2. **可中断** — 用户随时可以关闭终端，xiaoli 继续执行，回来能看到进度
3. **可回退** — 如果某一步执行失败，能回到安全状态（不丢用户代码）

### 3.2 执行计划数据模型

```typescript
interface ExecutionPlan {
  id: string;
  taskId: string;
  
  /** 计划摘要 */
  summary: string;         // 一句话描述
  estimatedSteps: number;  // 预估步数
  riskLevel: 'low' | 'medium' | 'high';
  
  /** 分阶段步骤 */
  phases: ExecutionPhase[];
  
  /** 前置条件检查 */
  prerequisites: {
    passed: boolean;
    checks: Array<{
      description: string;
      passed: boolean;
      errorMessage?: string;
    }>;
  };
  
  /** 生命周期 */
  status: 'draft' | 'confirmed' | 'running' | 'paused' | 'completed' | 'failed';
  createdAt: string;
  confirmedAt?: string;
  completedAt?: string;
}

interface ExecutionPhase {
  id: string;
  name: string;            // 「Phase 1: 问题定位」
  order: number;
  status: 'pending' | 'running' | 'paused' | 'completed' | 'failed';
  
  steps: ExecutionStep[];
  
  /** 进入条件（前一个 phase 必须完成） */
  entryConditions: string[];
  
  /** 退出条件（本 phase 完成的标准） */
  exitConditions: string[];
}

interface ExecutionStep {
  id: string;
  description: string;     // 「添加复合索引 (created_at, status, user_id)」
  tool: ToolName;          // read_file | write_file | terminal | git | database | ask_user
  input: Record<string, unknown>;
  expectedOutput: string;  // 「查询时间从 8.2s 降到 < 2s」
  
  /** 执行追踪 */
  status: 'pending' | 'running' | 'completed' | 'failed' | 'skipped';
  startedAt?: string;
  completedAt?: string;
  attempts: number;
  maxAttempts: number;     // 失败重试上限
  
  /** 回退操作 */
  rollback?: {
    description: string;
    tool: ToolName;
    input: Record<string, unknown>;
  };
  
  /** 失败时的替代方案 */
  fallback?: {
    description: string;
    tool: ToolName;
    input: Record<string, unknown>;
  };
}
```

### 3.6 中断与恢复

这是 xiaoli 最独特的场景——用户关上终端、去开会、回来继续。

```
关闭终端时:
  · 执行中的 Step → 标记为 paused（不终止，让它跑完）
  · 等待中的 Steps → 保持 pending
  · Plan 整体 → paused

重新打开终端时:
  · xiaoli status → 显示进度
  · xiaoli continue → 从 paused/resume 点继续

实现方式:
  · Plan 状态持久化到 SQLite
  · 每完成一个 Step 就写入数据库（不是内存状态）
  · continue 时从数据库读取 Plan，从第一个 pending Step 开始
```

## 四、追问与计划联动

追问和计划**不是串行两个阶段**，而是**交替进行**：

```
用户输入任务
  │
  ├─ 追问第一轮（宏观）──→ 用户回答 ──→ 初步计划（含假设）
  │
  ├─ 追问第二轮（微观）──→ 用户回答 ──→ 更新计划（假设减少）
  │
  ├─ 追问第三轮（收束）──→ 用户回答 ──→ 计划终稿
  │
  └─ 用户确认 ──→ 开始执行
```

为什么这样设计：
- 用户看着「初步计划」回答追问，会更有针对性
- 如果追问完了才出计划，用户可能发现「其实我想要的不是这样」，又要重新来
- 每轮追问后刷新计划，让用户感知到「越来越清晰」

### 5.3 终端追问收集逻辑

追问在终端里的交互用 `inquirer` 或自定义 readline 实现：

```typescript
// 伪代码：终端追问流程
async function collectAnswers(questions: Question[]): Promise<Answer[]> {
  const answers: Answer[] = [];
  
  for (const q of questions) {
    const answer = await askInTerminal(
      `[${q.category}] ${q.question}\n> `
    );
    
    if (answer === 'skip' || answer === '?' || answer === '不确定') {
      answers.push({ ...q, answer: '', resolved: false, hypothesis: null });
    } else {
      answers.push({ ...q, answer, resolved: true });
    }
  }
  
  return answers;
}

// 支持多答模式：「全部、3秒、后端」
function parseMultipleAnswers(raw: string): string[] {
  return raw.split(/[,，、]/).map(s => s.trim()).filter(Boolean);
}
```
