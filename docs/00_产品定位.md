# xiaoli（小李）—— 产品定位文档

> **一句话定义**：xiaoli 是一个住在终端里的 AI 同事。你交代任务、回答它的追问，然后它自己干活。你下班回来看结果。

---

## 1. 产品起源

### 1.1 这不是注意力管理系统

learntodo 试图做「判断系统 + 专注保护」，但根本问题不在「该做什么」或「怎么不被中断」——

**根本问题是你只有一个人。** 你同时是前端、后端、运维、客服。每天真正写代码的时间不到 4 小时，剩下的时间被沟通、排查、环境配置吃掉。

注意力管理的解法是「帮你更好地分配时间」。xiaoli 的解法是「**帮你扩展时间**」——当你回答完它的问题，它就是一个能独立干活的同事。

### 1.2 为什么叫 xiaoli（小李）

你叫别人干活时，喊的是「小李」「小张」。xiaoli 就是那个你每天喊来干活的人。不是助手、不是工具——是**同事**。

---

## 2. 用户画像

**唯一定位：独立开发者 / 中小团队软件从业者**

| 属性 | 描述 |
|---|---|
| 角色 | 全栈开发者（Vue + Java + Python） |
| 痛 | 一个人干三个人的活，时间永远不够 |
| 日常 | 早上打开电脑看记事本，挑最急的事做 |
| AI 使用习惯 | 已在使用 Claude Code / DeepSeek，但每次要手把手对话 |
| 技术环境 | 中国国内，DeepSeek API 可用 |

## 3. 核心工作流

### 3.1 两阶段模式

```
┌─────────────────────────────────────────────┐
│                PHASE 1: 问答阶段              │
│                                             │
│  你: xiaoli "修复ERP订单查询慢的问题"          │
│                                             │
│  xiaoli 提问（可能 5-15 个问题）：              │
│    - 慢到什么程度？有慢查询日志吗？              │
│    - 是哪个表？哪个接口？                       │
│    - 可以加索引吗？还是需要改 SQL？              │
│    - 需要改代码还是只改数据库？                  │
│    - 需要我给你方案先，还是直接动手？            │
│    - ...                                     │
│                                             │
│  你全部回答完毕。                              │
│                                             │
│  xiaoli 输出最终执行计划，等你确认。             │
│                                             │
│  你: 确认                                     │
│                                             │
├─────────────────────────────────────────────┤
│              PHASE 2: 自主执行阶段             │
│                                             │
│  xiaoli 开始干活：                            │
│    1. 搜索 orders 相关代码                     │
│    2. 分析查询逻辑                            │
│    3. 定位瓶颈                                │
│    4. 修改代码 / 添加索引                      │
│    5. 遇到 JDK 版本问题 → 自己解决              │
│    6. 跑测试                                  │
│    7. commit + push                          │
│    8. 汇报结果                                │
│                                             │
│  你合上屏幕去开会。回来后：                      │
│    xiaoli: "搞定了。查询从 8.2s → 0.3s。       │
│            已推到 fix/slow-order-query。       │
│            需要你 review 后合并。"               │
│                                             │
└─────────────────────────────────────────────┘
```

### 3.2 关键设计原则

**「先问完、后执行」** — 你不回答完所有问题，xiaoli 不写一行代码。这是和 Claude Code 最本质的区别：Claude Code 是边问边做的对话模式；xiaoli 是「需求评审通过 → 开发」的协作模式。

**「自己解决环境问题」** — JDK 版本不对、环境变量缺失、依赖冲突，xiaoli 自己处理，不回来问你。

**「持久记忆」** — xiaoli 记得你的项目、你的偏好、上次做到哪了。不需要每次都从零交代。

## 4. 三条核心命令

| 命令 | 行为 |
|---|---|
| `xiaoli "任务描述"` | 启动新任务 → 进入问答阶段 |
| `xiaoli status` | 显示当前进度：完成了什么、卡在哪 |
| `xiaoli continue` | 从中断点继续（你回来时用） |

不需要更多命令。你要的就是这三件事。

---

## 5. 自主级别（L1-L4）

| 级别 | 能力 | 是否需要确认 |
|---|---|---|
| L1 | 读取代码、搜索、分析 | 不需要 |
| L2 | 创建/修改文件 | 不需要（在计划内） |
| L3 | Git 操作（commit、push、创建分支） | 不需要（在计划内） |
| L4 | 数据库操作、服务重启、环境修改 | 不需要（在计划内） |
| **高危** | `rm -rf`、`DROP TABLE`、修改生产配置 | **立即打断，要求确认** |

核心逻辑：**计划内的操作自动执行，计划外的危险操作打断**。

## 6. AI 大脑

| 项目 | 决策 |
|---|---|
| API | DeepSeek API |
| 模型 | deepseek-v4-pro（主力）/ deepseek-v4-flash（轻量任务） |
| 部署 | 云端调用，本地不做推理 |
| 本地职责 | 文件读写、Shell 执行、Git 操作、权限控制 |

---

## 7. 技术形态

| 决策 | 理由 |
|---|---|
| **npm 包分发** | `npm install -g xiaoli`，和 Claude Code 一样的体验 |
| **Node.js + TypeScript** | npm 生态 + 类型安全 |
| **跨平台** | Windows / macOS / Linux 全支持 |
| **终端原生** | 不是网页、不是 APP、不是 IDE 插件 |

---

## 8. 和 Claude Code 的本质差异

| | Claude Code | xiaoli |
|---|---|---|
| 交互模式 | 对话式（你一句我一句） | **协作式（先问完再干活）** |
| 你不在时 | 停止等待 | **继续自主工作** |
| 任务粒度 | 单次对话 | **跨会话的持久任务** |
| 提问风格 | 边做边问 | **做完需求评审才开始写代码** |
| 环境处理 | 遇到问题抛给你 | **自己解决环境问题** |
| 返回体验 | 做完了对话就结束 | **你回来时主动汇报进度** |

xiaoli 不是「更聪明的 Claude Code」，而是「更像一个同事的 Claude Code」。

---

## 9. 不做的事

- ❌ 不做网页界面
- ❌ 不做手机 APP
- ❌ 不做 IDE 插件
- ❌ 不做「今日重点」「五步闭环」「梯度引擎」
- ❌ 不做「注意力分配」相关功能
- ❌ 不追求多端覆盖

**xiaoli 只有一个界面：终端。只有一种交互：对话。**
