## 二、逐品分析

### 2.1 DeepSeek-TUI（Hmbown / 29K star）

**一句话定位**：终端里的 DeepSeek 编程代理，Rust 编译成二进制，性能极致，是目前 DeepSeek 生态使用最广的 CLI 代理。

#### 交互流程

```
启动 deepseek → 选模式（Plan/Agent/YOLO）
  → 对话式交互
  → Plan 模式：只读探索，不动文件
  → Agent 模式：每步操作需用户确认
  → YOLO 模式：全自动执行，无确认
```

三种模式按风险分级：
- **Plan**：纯探索，零副作用。用于理解代码库。
- **Agent**：标准编程模式，每次工具调用弹出审批提示。
- **YOLO**：全自动执行，适合信任度高的重复性任务。

另有 `/model auto` 命令，自动选择模型和思维深度层级。

#### 技术架构亮点

| 模块 | 设计 | 价值 |
|---|---|---|
| **OS 级沙箱** | macOS Seatbelt / Linux Landlock / Windows Job Objects | L4 自主级别必备的安全底座 |
| **前缀缓存追踪** | 监控每轮对话缓存命中率，自动调整上下文结构 | 降成本 |
| **LSP 诊断集成** | 代码修改后自动跑 rust-analyzer / pyright / gopls | 免去手动检查 |
| **后台子代理** | 并行执行 10-20 个子代理 | 大规模重构时可以并行扫描多个模块 |
| **持久化任务队列** | 任务不随会话丢失，HTTP/SSE 运行时 API | 断点续传基础 |
| **成本追踪** | reasoning blocks 流式展示 + 每轮 token 统计 | 透明的成本感知 |

#### 可借鉴的设计

1. **OS 级沙箱**：xiaoli 要达到 L4 自主（可操作数据库/服务器），沙箱隔离是硬需求。DeepSeek-TUI 的 Seatbelt/Landlock 设计可以直接参考。
2. **子代理并发**：xiaoli 追问阶段需要并行分析多个文件、多个模块，子代理并发的设计模式可以直接复用。
3. **LSP 诊断集成**：xiaoli 改完代码后，自动跑 IDE 语法检查是低成本的质量保障。
4. **成本追踪**：xiaoli 如果"一直开着等用户回来"，失控的 API 调用是最大风险，必须内置成本追踪。

#### 关键差异：DeepSeek-TUI 一开始就干活，xiaoli 先问完再干活

DeepSeek-TUI 的 Plan 模式虽然只读，但它不是在"问问题"——它是在"理解代码库"。用户交代任务后，代理立即开始工具调用，边做边确认。xiaoli 的设计是：**用户回答完所有追问之前，不动一行代码。**

### 2.2 DeepSeek-Reasonix（esengine / 2.8K star）

**一句话定位**：DeepSeek 原生编程代理，核心卖点是「便宜到可以一直开着」——通过极致的前缀缓存优化，把月消费从 $150-250 压到 $12。

#### 交互流程

```
reasonix code my-project → 对话式
  → SEARCH/REPLACE 提案（工具调用返回修改方案）
  → 不直接写入磁盘
  → /apply 命令才落盘

reasonix chat → 纯对话，无文件权限

reasonix run "task" → 一次性执行，结果输出到 stdout，不进入持久会话
```

三种模式：
- **code**：文件系统 + Shell 工具，需要审批
- **chat**：纯对话，零权限
- **run**：一次性任务，无状态

核心安全设计：「提案生成但不落盘，`/apply` 才写」——即使用户不盯着屏幕，也不会产生意外副作用。

#### 技术架构：四根柱子

| 柱子 | 解决的问题 | 具体做法 |
|---|---|---|
| **Pillar 1: Cache-First Loop** | DeepSeek 上下文缓存命中率常 < 20%，大量 token 被重复计费 | 上下文切三段：不可变前缀（系统提示词+工具定义）+ 只追加日志（对话历史）+ 每轮重置草稿（当前工具调用结果）。保证前缀永不变化，缓存命中率 99.82% |
| **Pillar 2: Tool-Call Repair** | DeepSeek 偶发输出损坏的 JSON（括号不匹配、截断、嵌套错误） | 四道修复工序：flatten（复杂 schema 打平简化）、scavenge（从 reasoning 文本中捞回丢失字段）、truncation（补全截断的 JSON）、storm（拦截重复调用） |
| **Pillar 3: Cost Control** | 活跃用户月消费 $150-250，不可持续 | flash-first 默认（V3 便宜模型先跑）、轮末自动压缩结果、`/pro` 单轮升级到 V4、失败信号自动升级模型 |
| **Pillar 4: Parallel Dispatch** | 工具调用的串行浪费——每个工具调用等前一个完成 | 同类只读工具并行跑（Promise.allSettled），写入工具按序串行 |

#### 实测数据

- 上下文三段式分区后，435M 输入 token，**99.82% 缓存命中**
- 同上场景：实际支出 $12，而非按标准计费的 $61
- 减价 80%

#### 可借鉴的设计

1. **上下文三段式分区（Pillar 1）**：xiaoli 的追问阶段是长对话，如果在追问过程中频繁重排系统提示词位置，缓存命中率会暴跌。必须把系统提示词锁定在前缀不变部分。这是所有竞品中最成熟的前缀缓存设计，**xiaoli 必须采纳**。
2. **成本控制（Pillar 3）**：xiaoli "先问完、后执行"的模式，追问阶段可能产生大量 token。flash-first + 自动压缩 = 必须做。执行阶段：用户离开后 xiaoli 自主干活，如果遇到循环或死锁，cost control 是生命线。
3. **Tool-Call Repair（Pillar 2）**：中国用户访问 DeepSeek API 可能遇到网络波动导致输出截断。这四道修复工序可以作为 xiaoli 的韧性层。
4. **安全写入（提案→`/apply`→落盘）**：xiaoli 的执行计划审批后，具体文件操作可以沿用这个模式。不是直接 write_file，而是 propose_edit → 用户回来 review → 批量 apply。

#### 关键差异：Reasonix 是交互式的，xiaoli 是自主式的

Reasonix 每轮对话都是"用户一句、代理一句"的交互循环。xiaoli 是"用户交代完整任务→代理问完所有问题→用户一次性批准→代理自主执行数小时"。Reasonix 的设计假设用户在线，xiaoli 的设计假设用户离线。

### 2.3 Langcli（LangcliTeam / 156 star）

**一句话定位**：Claude Code 的"互操作版"——换个壳接各种模型，100% 兼容 Claude Code 的 `.claude` 配置、skills、slash commands。

#### 交互流程

```
langcli → 和 Claude Code 完全一样的交互体验
  → /model 随时切换模型，不中断上下文
  → 通过 LangRouter 路由请求到不同后端
```

用户体验和 Claude Code 完全一致，唯一差异是可以在对话中途切换模型。比如用 DeepSeek 做探索，遇到棘手问题切到 Claude 解决，再切回来。

#### 技术架构亮点

| 模块 | 设计 |
|---|---|
| **运行时** | Bun，打包成单文件 |
| **权限系统** | 6300+ 行代码，plan / auto / manual 三种模式 + YOLO 分类器 |
| **功能丰富度** | 60+ feature flag，代理工具、后台任务、定时任务、工作树隔离 |
| **模型切换** | LangRouter 抽象层，运行时切换不丢上下文 |

#### 可借鉴的设计

1. **权限分级系统**：6300 行权限代码代表了成熟的审批流程设计。xiaoli 的「高危操作需手打确认短语」可以从这个设计里提取模式。
2. **多模型切换开关**：xiaoli 虽然用 DeepSeek，但留下备用模型的接口是零成本的前瞻设计。当 DeepSeek 服务不稳定时，可以切到备用。
3. **工作树隔离**：不同任务使用独立的文件系统视图，避免相互污染。

#### 关键差异：Langcli 是 Claude Code 换皮，xiaoli 是全新交互范式

Langcli 没有重新设计交互流程，它复制了 Claude Code 的交互模式。xiaoli 「先问→确认→自主执行」的范式与 Claude Code 完全不同，不是换皮能实现的。

## 三、三个竞品的共同架构模式

尽管产品定位和交互流程各有不同，三个竞品在架构层面共享以下模式。这些是 chat→code 代理领域的「事实标准」，xiaoli 也应该遵循：

### 3.1 代理循环（Agent Loop）

```
用户输入 → 构建上下文 → 发给 LLM → 流式接收响应
  → 解析工具调用列表 → 执行工具
  → 结果格式化后写回上下文 → 下一轮循环
```

三个项目都是这个循环。区别在于：
- **DeepSeek-TUI**：循环中每次工具调用暂停等用户确认（Agent 模式）
- **Reasonix**：循环中生成提案但不落盘，用户 `/apply` 才是真正的执行
- **Langcli**：循环与 Claude Code 完全一致

xiaoli 的循环会变为两层：
```
外层：追问循环（用户在线）
内层：执行循环（用户离线）
```

### 3.2 提示词分区（Prompt Partitioning）

```
┌──────────────────────────────┐
│  不可变前缀                   │
│  系统提示词 + 工具定义 + 规范  │
├──────────────────────────────┤
│  追加日志                     │
│  对话历史（只增不减）          │
├──────────────────────────────┤
│  本轮草稿                     │
│  当前轮工具调用及结果（每轮替换）│
└──────────────────────────────┘
```

这是 Reasonix Pillar 1 的核心设计，DeepSeek-TUI 也采用类似分区。这是 DeepSeek 上下文缓存优化的最佳实践。xiaoli 必须采用此设计（见 §5.1）。

### 3.3 安全闸门（Approval Gate）

```
工具调用 → 生成操作提案 → 展示 diff → 等待审批 → 执行 → 记录审计日志
```

三个项目都有某种形式的审批闸门：
- DeepSeek-TUI：Agent 模式下逐工具审批，YOLO 模式跳过
- Reasonix：`/apply` 批量审批
- Langcli：6300 行权限系统，支持计划/自动/手动三种审批模式 + YOLO 分类器

xiaoli 的创新点：**审批不在执行阶段发生，而是在计划阶段一次性完成**（见 §4）。

### 3.4 工具注册表（Tool Registry）

```
每个工具：
  - name: string           // 工具名称（LLM 选工具的依据）
  - description: string    // 功能描述（Prompt Injection 的关键输入）
  - schema: JSON Schema    // 参数定义（约束 LLM 输出格式）
  - handler: async (params) => Result  // 执行函数

注册 → LLM 选择工具 → 调度执行 → 结果格式化 → 写回上下文
```

三个项目都有统一的工具注册机制。xiaoli 的差异在于需要额外区分：
- **计划阶段可用工具**：只读（搜索、读取、分析）
- **执行阶段可用工具**：全能力（读写、执行、Git、数据库）

## 四、xiaoli 与竞品的关键差异

### 4.1 核心差异：三个竞品都没做到的事情

| | 三个竞品 | xiaoli |
|---|---|---|
| **开始干活时机** | 立即开始，边说边做 | 问完所有问题 → 用户确认全部计划 → 才开始 |
| **用户不在时** | 停在那等着。用户关闭终端 = 会话结束 | **继续自主干活**。回来时汇报进度 |
| **确认粒度** | 每步 / 每轮 / 批量都可能需要确认 | **一次性确认全部执行计划，然后自主执行到底** |
| **环境自理** | 遇到环境问题（JDK 版本不对、缺少依赖）一般停住等用户处理 | **自行诊断并修复环境问题**（在确认计划阶段已明确环境约束） |
| **产品形态心智** | 「和你一起写代码的助手」 | 「替你干活的同事」 |

### 4.2 为什么竞品没做「先问后做」

因为三个项目都是从「聊天机器人 + 工具调用」这个范式自然演化出来的——用户说一句，它做一步。这是一个本地最小化的产品演化路径。

xiaoli 的「先问后做」需要额外的系统设计：

1. **追问生成器**：分析用户输入，穷举所有歧义点。这不是工具调用，而是在工具调用之前增加一个独立的推理阶段。
2. **执行计划引擎**：将用户的回答和 AI 的理解编译成一个可验证、可追溯的结构化执行计划。
3. **断点续传机制**：用户离开期间自主执行，需要状态持久化 + 进度汇报 + 安全闸门。
4. **异步进度汇报**：用户回来时不只看日志，而是"已完成 Step 3/7，卡在 Step 4，原因是……"

这三个项目没有这些模块，因为它们的架构假设了用户始终在线。

### 4.3 xiaoli 的垂直壁垒

这不是一个功能差异——这是一个**产品范式的差异**。xiaoli 要解决的不是「怎么更聪明地写代码」，而是「怎么把写代码这件事从用户的时间线里拿掉」。

这决定了 xiaoli 的对手不是 DeepSeek-TUI，而是「用户自己的时间」。只要 xiaoli 能让用户一天多出 2-3 小时有效时间，用户就会用。

## 五、对 xiaoli 技术选型的具体建议

基于竞品分析，以下是必须采纳和强烈建议的设计决策：

### 5.1 必须采纳（来自竞品验证的最佳实践）

| # | 设计决策 | 来源 | 理由 |
|---|---|---|---|

| 2 | **安全闸门：提案→审批→落盘** | Reasonix + 三者共同模式 | xiaoli L4 自主级别下，直接写文件太危险。计划审批后，执行阶段使用 `propose_edit → 批量 apply` 模式。 |

| 6 | **LSP 诊断集成** | DeepSeek-TUI | 代码修改后自动跑语言服务器诊断，低成本质量保障。 |

### 5.2 强烈建议（来自竞品的有价值设计）

| # | 设计决策 | 来源 | 理由 |
|---|---|---|---|
| 7 | **OS 级沙箱** | DeepSeek-TUI | L4 自主操作数据库和服务器，沙箱隔离是硬需求。Linux Landlock 是首选方案。 |
| 8 | **子代理并发** | DeepSeek-TUI | 追问阶段并行分析多个模块、执行阶段并行跑测试和 lint。 |
| 9 | **持久化任务队列** | DeepSeek-TUI | xiaoli 的断点续传依赖持久化。任务不能随会话丢失。 |
| 10 | **多模型切换接口** | Langcli | DeepSeek 服务不稳定时可以切备用模型。零成本的前瞻设计。 |

### 5.3 xiaoli 独有模块（竞品没有的）

| # | 模块 | 说明 |
|---|---|---|
| 11 | **追问生成器** | 分析用户输入 → 穷举歧义点 → 按重要性排序 → 逐一提问 → 用户回答 → 生成完整需求文档 |
| 12 | **执行计划引擎** | 将完整需求编译为结构化执行计划（步骤列表 + 每步预期输出 + 依赖关系 + 验证方法） |
| 13 | **断点续传** | 用户关闭终端后后台继续执行。记录每步状态（pending/in_progress/completed/blocked），回来时可用 `xiaoli status` 查看 |
| 14 | **异步进度汇报** | `xiaoli status` 返回结构化进度：「Step 3/7 已完成 ✓，Step 4 正在执行中，预计还需 15 分钟」而非原始日志堆砌 |

## 六、竞品启示总结

### 6.1 三条最重要的启示

1. **成本不是优化项，是生命线。** Reasonix 把月消费从 $150+ 压到 $12，靠的不是砍功能，而是上下文缓存优化。xiaoli 如果用户「一直开着等它干活」，成本控制必须从第一天做起。

2. **安全不是功能，是产品形态的前提。** L4 自主意味着 xiaoli 可以操作文件系统和数据库。没有沙箱和安全闸门，用户不敢让它自己干活。安全决定了 xiaoli 的信任模型。

3. **交互范式才是壁垒，不是模型能力。** 三个竞品用同一个大模型（DeepSeek V4），但交互体验完全不同。xiaoli 的「先问后做」范式是真正的差异化——它不依赖于比竞品更聪明的大模型，而依赖于更符合用户真实工作流的设计。

### 6.2 一条反直觉的结论

**xiaoli 的第一个版本不应该追求功能多，而应该追求「追问质量」极致高。**

因为用户的第一次体验不是「xiaoli 帮我写了多少代码」，而是「xiaoli 问的问题让我觉得它真的懂了我要做什么」。追问质量决定了用户是否敢把执行权交出去。

如果 xiaoli 问了 5 个无意义的问题然后开始瞎写代码，用户不会再用了。如果 xiaoli 问了 3 个用户自己都没想到但确实需要确认的问题，即使执行慢一点，用户也会等。
