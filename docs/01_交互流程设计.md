# xiaoli（小李）—— 交互流程设计

> 本文档定义 xiaoli 和你之间的每一次对话如何发生。

---

## 1. 问答阶段（Phase 1）

### 1.1 触发

```bash
xiaoli "修复ERP订单查询慢的问题，用户说导出报表要等10秒以上"
```

xiaoli 解析任务后，不写任何代码，开始提问。

### 1.2 提问规则

**提问策略：先宽后窄，5-15 个问题。**

```
第一轮（广度）：搞清楚边界
  1. 是所有订单查询都慢，还是特定条件慢？
  2. 数据库用的什么？版本？
  3. 这个功能是线上在生产用，还是测试环境？

第二轮（深度）：钻进细节
  4. 慢查询日志能拿到吗？
  5. 有没有 EXPLAIN 的结果？
  6. 表大概多少行数据？
  7. 有没有已经建了的索引？列一下？

第三轮（执行偏好）：确定怎么干
  8. 索引可以直接加生产库吗，还是要先在测试环境验证？
  9. 需要我同时优化 SQL 本身吗？
  10. 修改后需要跑什么测试确认不引入新问题？
```

### 1.3 你在回答时

```
你: 1. 主要是按日期范围 + 状态筛选的时候慢，单条查询不慢
你: 2. MySQL 8.0
你: 3. 线上生产
你: 4. 慢查询日志有，我可以发给你
你: 5. 还没跑 EXPLAIN
你: 6. orders 表大概 2000万行，order_items 大概 5000万行
你: 7. orders 表有主键索引和 user_id 索引
你: 8. 先在测试环境加索引，确认没问题再上生产
你: 9. 可以，先把明显的问题改了
你: 10. 跑已有的单元测试就行，不需要额外写
```

### 1.4 xiaoli 输出执行计划

你回答完所有问题后，xiaoli 输出最终计划：

```
xiaoli: 好的，我理解的执行计划如下：

  步骤 1: 连接测试数据库，对 orders 表运行 EXPLAIN
  步骤 2: 分析慢查询日志，定位瓶颈 SQL
  步骤 3: 在测试库添加复合索引 (created_at, status)
  步骤 4: 重写导出查询 SQL，拆分为分批查询
  步骤 5: 跑现有单元测试确认不引入回归
  步骤 6: 生成生产库执行脚本（你手动确认后执行）

  预计耗时：约 30-50 分钟

  确认开始？
```

你只需回复一个字：**「确认」**。

## 2. 自主执行阶段（Phase 2）

### 2.1 xiaoli 的行为

确认之后，xiaoli 开始干活。你不需要盯着它。

它的操作日志会持续写入，但你不需要实时看。你关掉终端去开会、去写别的代码、去吃午饭。

### 2.2 你回来时

```bash
$ xiaoli status
```

```
xiaoli: 任务「修复ERP订单查询慢」进度：

  ✅ 步骤1: EXPLAIN 分析完成 — 确认 full table scan
  ✅ 步骤2: 慢查询日志分析完成 — 定位到 3 条问题 SQL
  ✅ 步骤3: 测试库索引添加完成
  ✅ 步骤4: SQL 重写完成（3 条 → 分批查询 + JOIN 优化）
  ⏳ 步骤5: 单元测试跑了一半，account 模块全过，order 模块进行中
  ⬜ 步骤6: 生产脚本等你确认

  已完成 4/6 步，正在执行第 5 步。
```

### 2.3 如果卡住了

```
xiaoli: 步骤5 遇到问题：

  数据库迁移脚本依赖 flyway 7.x，但你的测试环境装的是 flyway 9.x。
  我已经尝试了自动降级，但 flyway 7 和你的 MySQL 8.0.32 不兼容。

  我正在尝试的替代方案：
  → 不用 flyway，直接通过 JDBC 执行 DDL 语句（ALTER TABLE ADD INDEX）

  这个方案可以继续吗？还是你有其他偏好？

  按 Enter 确认，或输入新的指示。
```

这是 xiaoli 的「环境自愈」能力：它先自己尝试解决，只有自己解决不了了才问你。

## 3. 中断与恢复

### 3.1 中断场景

你正在会议室里，xiaoli 在后台跑着。你不需要做任何事——xiaoli 继续工作。

### 3.2 你回来继续

```bash
$ xiaoli continue
```

xiaoli 读取上次中断的状态，从中断点继续执行。

如果上一个任务已经完成了，`xiao continue` 等同于 `xiao status`。

### 3.3 新任务打断旧任务

```bash
$ xiaoli "紧急：支付回调接口报500，先处理这个"
```

```
xiaoli: 当前有一个进行中的任务「修复ERP订单查询慢」（已完成 4/6 步）。

  新任务「修复支付回调接口500」要：
    A. 暂停当前任务，先做这个
    B. 排队，等当前任务完成再做
    C. 放弃当前任务

  选哪个？
```

## 4. 对话风格

### 4.1 xiaoli 的口吻

- **简洁**：不废话。不是为了「看起来聪明」，是为了让你快速回答。
- **结构化**：问题编号、步骤编号。你知道它想问什么。
- **非拟人化**：不自称「我理解了」「让我想想」。直接做事。
- **中文**：所有对话用中文。

### 4.2 举例

**好：**
```
xiaoli: 订单表大概多少行？有分表吗？
```

**差：**
```
xiaoli: 我理解你遇到了一个性能问题，让我来帮你分析一下。
       首先我需要了解一些基本信息，请问你的订单表数据量大概在什么级别呢？
```

## 5. 安全边界

### 5.1 高危操作拦截

以下操作**无论是否在计划内**都必须打断要求确认：

| 操作 | 检测方式 |
|---|---|
| `rm -rf` / 删除目录 | 命令模式匹配 |
| `DROP TABLE` / `DROP DATABASE` | SQL 解析 |
| `git push --force` | Git 命令解析 |
| 修改生产数据库连接配置 | 配置文件路径匹配 |
| `chmod 777` / 权限放宽 | 命令模式匹配 |
| 超过 100 行的文件删除 | 文件操作监控 |

### 5.2 确认话术

```
xiaoli: ⚠️  高危操作拦截

  即将执行：DROP INDEX idx_user_id ON orders
  影响范围：生产数据库 orders 表，2000万行
  操作不可回滚。

  输入 "确认执行 DROP INDEX" 以继续。
```

必须输入完整确认短语，不接受 Enter 默认确认。

## 6. 记忆系统

### 6.1 xiaoli 记住什么

```
项目级记忆（每个 Git 仓库独立）：
  - 项目技术栈（Vue 3 + Java + Python）
  - 数据库类型和版本
  - 常用分支命名规范
  - 上次任务进展

用户级记忆（跨项目）：
  - 编码偏好（SQL 风格、命名习惯）
  - AI 沟通习惯（详细解释 vs 直接给方案）
  - 时间相关（通常你在哪个时段可以回应）
```

### 6.2 xiaoli 不记住什么

- ❌ 代码具体内容（每次重新读取）
- ❌ 数据库连接密码
- ❌ 密钥和 Token
